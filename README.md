# Portfolio

서로 다른 계층과 제약을 가진 시스템을 공통된 설계 기준으로 정리해 온 개발자입니다.  
저는 기능을 빠르게 추가하는 것보다, 요구사항이 바뀌었을 때 수정 지점이 분명하고 영향 범위가 예측되는 구조를 만드는 일을 더 중요하게 생각합니다.

이 기준은 프로젝트를 거치며 점차 선명해졌습니다.  
Ocean Cloud에서는 복잡한 화면과 상태를 정리하는 구조 감각을 익혔고,  
Key Fetcher에서는 변경 이유를 기준으로 정책과 책임을 분리하는 설계 기준을 세웠고,  
Relion에서는 그 기준을 시스템 전체의 경계와 독립성으로 확장했습니다.

## 대표 프로젝트

| 프로젝트 | 요약 |
|---|---|
| [Relion](./embedded/relion/README.md) | 여러 종류의 임베디드 장비에 이식 가능한 REST API 서버입니다.<br/>HTTP 엔진이 바뀌어도 어플리케이션 로직 전체를 다시 수정하지 않도록, HTTP 처리와 요청 흐름, 어플리케이션 연결의 책임을 나누어 설계·구현했습니다. |
| [Key Fetcher](./embedded/key-fetcher/README.md) | AP 관리 프로토콜 암호화 키 교환 시스템입니다.<br/>재시도, 타임아웃, 로그 처리, 상태 전이를 나누어 설계하고, 반복된 스펙 변경에도 영향이 있는 지점만 수정할 수 있도록 정리했습니다. |
| [Ocean Cloud](./web/oceancloud/README.md) | 서버 정의 기반 동적 입력 및 데이터 탐색 플랫폼입니다.<br/>입력 항목이 바뀔 때마다 화면 코드를 전체 수정하지 않도록, 상태 관리와 렌더링 책임을 나누고 수정 지점을 줄이는 방향으로 구현했습니다. |

## 작업 방식

- 요구사항이 바뀌었을 때 어디를 고쳐야 하는지가 바로 보이도록 책임과 경계를 먼저 나눕니다.
- 외부 의존성이나 입출력 방식이 안쪽 로직까지 바로 번지지 않도록 설계합니다.
- 장애가 났을 때 흐름을 따라가며 원인을 좁힐 수 있도록 상태 변화와 실패 지점을 드러냅니다.
- 설계에서 정한 책임과 경계가 코드와 문서 양쪽에 같은 방향으로 드러나도록 함께 관리합니다.
