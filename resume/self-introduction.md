# 자기소개

서로 다른 계층과 제약을 가진 시스템을 공통된 설계 기준으로 정리해 온 개발자입니다.  
저는 기능을 빠르게 추가하는 것보다, 요구사항이 바뀌었을 때 수정 지점이 분명하고 영향 범위가 예측되는 구조를 만드는 일을 더 중요하게 생각합니다.  
임베디드와 웹을 모두 경험했지만, 관심의 중심은 특정 기술 자체보다 시스템의 경계, 책임, 상태 흐름을 어떻게 나눌지에 있습니다.

임베디드 개발 경험은 운영체제, 네트워크, 동시성, 장애 대응처럼 시스템의 기초를 이해하는 기반이 되었습니다.  
특히 AP 개발에서는 L2/L3 제어부터 L7 관리 인터페이스까지 함께 다루며, 서로 다른 계층이 실제로 어떻게 연결되고 어디서 영향이 번지는지를 계속 경험했습니다.  
이 경험을 통해 기능 구현 자체보다, 수정과 교체가 일어날 때 영향 범위를 통제할 수 있는 구조를 먼저 생각하는 기준을 갖추게 되었습니다.

## 핵심 역량

제가 가장 자신 있는 부분은 복잡한 요구사항을 바로 코드로 옮기기보다, 먼저 변경 가능성이 큰 지점과 고정되어야 할 지점을 나누는 일입니다.  
기능이 추가되더라도 기존 코드 전체를 다시 건드리지 않도록 책임을 분리하고, 외부 의존성이 안쪽 로직까지 직접 들어오지 않도록 경계를 두는 방식을 선호합니다.  
이 기준을 통해 수정 범위를 예측 가능하게 만들고, 장애가 발생했을 때도 상태 변화와 실패 지점을 따라가며 문제를 좁힐 수 있는 구조를 만드는 데 강점이 있습니다.

또한 저는 설계와 구현, 문서를 따로 보지 않습니다.  
문서는 설명을 덧붙이는 자료가 아니라, 무엇을 어디에 둘지 결정한 내용을 남기는 설계 기록이라고 생각합니다.  
요구사항 정리, 설계, 의사결정 기록, 구현이 서로 다른 방향으로 흩어지지 않도록 맞추는 일 역시 중요하게 생각합니다.

## 대표 사례

AP 관리 프로토콜 암호화 프로젝트에서는 예외 상황이 많은 키 교환 로직을 재시도, 타임아웃, 상태 전이, 로그 처리로 나누어 정리했습니다.  
그 결과 스펙이 바뀌더라도 전체 흐름을 다시 뜯어고치기보다, 영향이 있는 정책과 흐름만 수정하는 방식으로 대응할 수 있었습니다.  
이 경험을 통해 수정 지점을 먼저 분리해 두는 설계가 실제 대응 속도와 유지보수성에 직접 연결된다는 점을 확인했습니다.

REST API 서버 개발에서는 HTTP 처리, 라우팅, 요청 변환, 어플리케이션 연결의 책임을 분리하고, 외부 엔진 의존성을 경계 뒤로 감췄습니다.  
그래서 HTTP 엔진이나 입출력 방식이 바뀌더라도 어플리케이션 로직까지 수정이 번지지 않도록 정리할 수 있었습니다.  
이 프로젝트는 기능 구현 자체보다, 변경이 들어와도 수정 범위를 좁게 유지할 수 있는 구조를 세우는 작업에 가까웠습니다.

Ocean Cloud 프로젝트에서는 동적 입력 화면과 데이터 탐색 기능을 구현하면서, 입력 항목이나 화면 구성이 바뀔 때마다 UI 전체를 다시 수정하지 않도록 서버 정의 기반으로 화면을 구성했습니다.  
이 과정에서 상태 관리와 렌더링 책임을 나누고, 화면 확장 시 어떤 부분을 수정해야 하는지가 드러나도록 구조를 정리하는 데 집중했습니다.  
이 경험을 통해 사용자 인터페이스 역시 화면을 그리는 기술보다, 변경이 들어왔을 때 수정 지점이 분명한 형태로 만드는 일이 중요하다는 기준을 갖게 되었습니다.

## 일하는 방식

저는 구현에 바로 들어가기보다, 먼저 어떤 책임을 어디에 둘지부터 정리하는 편입니다.  
무엇을 만들지보다 무엇이 어디까지 영향을 줄 수 있는지를 먼저 정해야 이후의 수정 비용과 협업 비용을 줄일 수 있다고 보기 때문입니다.  
그래서 설계 단계에서는 정책, 책임, 의존 방향을 먼저 정리하고, 구현 단계에서는 그 경계가 흐려지지 않도록 계속 확인합니다.

리뷰를 할 때도 같은 기준을 유지합니다.  
스타일 차이보다 책임이 섞여 있는지, 수정 한 건이 불필요하게 여러 계층으로 번지는지, 외부 포맷이나 구현 세부사항이 경계를 넘어 들어왔는지를 더 중요하게 봅니다.  
또한 반복되는 작업은 문서와 템플릿으로 정리해 해석 차이를 줄이고, 기능 개발과 구조 변경을 섞지 않으려 합니다.

## 지향점

지금까지 세워 온 설계 기준을 더 큰 시스템과 협업 맥락에서도 유지하고 확장하는 방향으로 성장하고 싶습니다.

구체적으로는 두 가지를 이어가고 싶습니다.
하나는 분산 환경이나 더 큰 규모의 시스템에서도 수정 범위를 예측 가능하게 유지하는 설계 경험을 쌓는 것입니다.
다른 하나는 설계 판단을 팀 안에서 기준으로 공유하고, 리뷰와 문서 흐름 전체에 녹여 내는 역할을 더 잘하는 것입니다.

임베디드와 웹을 함께 경험하면서 각 계층이 어떻게 연결되는지를 본 시야는, 앞으로 어떤 환경에서 작업하더라도 기준을 잃지 않는 바탕이 될 것이라 생각합니다.
