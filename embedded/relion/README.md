# Relion

Relion은 여러 종류의 임베디드 장비와 리눅스 기반 환경에 포팅 가능한 REST API 서버입니다.  
HTTP 엔진이나 실행 환경이 바뀌더라도, 영향이 필요한 경계만 수정하고 어플리케이션 로직은 그대로 둘 수 있도록 설계했습니다.

이 프로젝트에서 중요했던 것은 REST API 기능을 만드는 것 자체보다, 외부 엔진 차이와 실행 환경의 제약이 전체 코드로 번지지 않게 막는 일이었습니다.  
그래서 Core, Framework, Application의 책임을 나누고, 각 계층 사이에는 Adapter를 두어 연결 방식을 분리했으며, 어플리케이션 내부에서는 Infrastructure를 통해 외부 시스템 의존성이 안쪽 로직으로 직접 들어오지 않도록 정리했습니다.

## 핵심 내용

- Core / Framework / Application 3계층으로 나누어 HTTP 엔진과 어플리케이션 로직의 영향을 분리했습니다.
- 계층 사이에는 Adapter를 두어 연결 방식을 분리하고, 상위 계층이 하위 구현 세부사항에 직접 끌려가지 않도록 정리했습니다.
- 어플리케이션 내부에서는 Infrastructure를 통해 외부 시스템과의 의존성을 격리해, 도메인과 유스케이스까지 영향이 번지지 않도록 했습니다.
- layer-check를 통해 계층 위반을 빌드 단계에서 검출할 수 있도록 했습니다.

## 왜 이 프로젝트가 중요했는가

Relion은 일반적인 웹 서비스 구조를 그대로 가져와 만들 수 있는 프로젝트가 아니었습니다.  
임베디드 장비와 리눅스 기반 환경에서는 라이센스, HTTP 엔진 선택, 시스템 자원, 실행 환경 차이 같은 제약이 먼저 존재했고, 이 제약을 무시하면 기능을 추가할수록 수정 범위가 쉽게 커질 수 있었습니다.

이 프로젝트의 핵심 질문은 다음과 같았습니다.

- HTTP 엔진이 바뀌어도 어플리케이션 로직을 그대로 둘 수 있는가
- 실행 환경 차이가 생겨도 시스템 전체 수정으로 번지지 않게 만들 수 있는가
- 요청 처리 흐름과 비즈니스 로직을 분리해 계층을 유지할 수 있는가
- 이런 규칙을 문서가 아니라 빌드 단계에서 검증할 수 있는가

Relion은 이 질문에 구조로 답하기 위해 만든 프로젝트입니다.

## 결과

- HTTP 엔진 교체 대응 1일
- 어플리케이션 수정 없이 교체 검증
- 엔드포인트 20개 이상 확장
- layer-check를 통한 계층 위반 자동 검출

## 제 역할

이 프로젝트에서는 요구사항 분석부터 아키텍처 설계, Framework 구현, Adapter 경계 설계, 어플리케이션 계층 구성, layer-check 구축, 이식성 검증까지 전체 흐름을 단독으로 진행했습니다.

중요하게 본 것은 기능 수를 늘리는 것보다, 변경이 들어왔을 때 어디를 고쳐야 하는지가 분명하고 수정 영향이 다른 계층으로 쉽게 번지지 않도록 기준을 세우는 일이었습니다.
