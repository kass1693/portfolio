# Overview

## 프로젝트 개요

Key Fetcher는 AP 관리 프로토콜 암호화 과정에서 필요한 키 교환 시스템입니다.  
이 프로젝트의 핵심은 키를 주고받는 기능 자체보다, 예외 상황이 많은 흐름을 어떻게 나누고 스펙 변경이 들어와도 수정 범위를 좁게 유지할 것인가에 있었습니다.

실제 운영 환경에서는 재시도 조건, 타임아웃 기준, 상태 전이, 실패 처리 방식이 서로 얽혀 있었고, 프로토콜 변경도 반복적으로 발생했습니다.  
이런 상황에서 키 교환 로직을 한 덩어리로 두면 작은 수정도 전체 흐름으로 번지기 쉬웠기 때문에, 처음부터 변경 지점을 나누는 방향으로 설계를 진행했습니다.

## 어떤 문제가 있었는가

이 프로젝트에서 다뤄야 했던 문제는 단순한 요청-응답 흐름이 아니었습니다.  
키 교환은 여러 단계의 상태를 따라 진행되며, 중간에 응답 지연, 실패, 재시도, 타임아웃 같은 예외가 자주 개입했습니다.  
여기에 운영 중 스펙 변경까지 반복되면서, 기능을 먼저 구현하고 나중에 수정하는 방식으로는 대응 속도와 안정성을 함께 가져가기 어려운 상황이었습니다.

특히 다음과 같은 점이 중요했습니다.

- 상태에 따라 다음 동작이 달라졌습니다.
- 재시도와 타임아웃 기준이 자주 바뀔 수 있었습니다.
- 실패했을 때 어디서 끊겼는지 추적할 수 있어야 했습니다.
- 로그가 단순 기록이 아니라 운영 중 원인 파악의 기준이 되어야 했습니다.

결국 이 프로젝트의 핵심 문제는 키를 교환하는 기능 그 자체보다, 변경과 예외가 많은 흐름을 어떻게 관리 가능한 형태로 나눌 것인가였습니다.

## 어떻게 접근했는가

이 프로젝트에서는 키 교환 전체를 하나의 큰 로직으로 두지 않았습니다.  
대신 상태 전이, 재시도, 타임아웃, 로그 처리처럼 자주 바뀌거나 운영 중 중요하게 보는 지점을 나누어 설계했습니다.  
목표는 기능을 화려하게 만드는 것이 아니라, 스펙이 바뀌었을 때 전체 로직을 다시 건드리지 않고 영향이 있는 부분만 수정할 수 있게 만드는 것이었습니다.

예를 들어 재시도 조건이 달라지면 재시도 판단이 있는 부분만 수정하고, 타임아웃 기준이 달라지면 시간 관련 처리 지점만 수정할 수 있어야 했습니다.  
또한 장애가 발생했을 때는 상태 흐름과 로그를 기준으로 어느 단계에서 문제가 생겼는지 좁혀갈 수 있어야 했습니다.

이런 식으로 흐름을 나누어 두면, 변경 대응과 운영 대응을 같은 기준 위에서 다룰 수 있습니다.  
한 번에 모든 문제를 해결하려 하기보다, 자주 바뀌는 부분과 추적이 필요한 부분을 먼저 분리하는 것이 이 프로젝트의 핵심 설계 판단이었습니다.

## 무엇이 달라졌는가

이렇게 정리한 뒤에는 스펙 변경이 들어왔을 때 전체 키 교환 로직을 다시 보는 대신, 영향이 있는 정책과 단계 중심으로 수정 범위를 좁힐 수 있었습니다.  
이 차이는 실제 대응 속도에 직접 연결되었습니다.

또한 운영 중 문제가 발생했을 때도 단순히 실패 로그만 남기는 것이 아니라, 상태 흐름을 따라가며 어느 단계에서 어떤 이유로 실패했는지 추적할 수 있게 되었습니다.  
즉, 이 프로젝트는 기능을 추가하는 일보다 변경과 장애를 다루는 비용을 줄이는 쪽에서 더 큰 효과가 있었습니다.

## 결과

이 프로젝트에서는 반복된 스펙 변경 16건에 대응했고, 평균 1일 내 수정 반영이 가능했습니다.  
또한 6개월 이상 필드 테스트를 운영하면서 실제 환경에서 변경 대응과 장애 추적이 가능한 형태로 흐름을 유지했습니다.

정리하면, Key Fetcher의 핵심 성과는 키 교환 기능을 구현한 것 자체보다 다음에 있었습니다.

- 스펙 변경 시 전체 로직이 아니라 영향이 있는 지점만 수정할 수 있게 만든 점
- 장애 발생 시 상태 흐름과 로그를 기준으로 원인을 좁힐 수 있게 한 점
- 운영 중 반복되는 변경과 예외를 같은 설계 기준으로 다룰 수 있게 한 점

## 제 역할

이 프로젝트에서는 요구사항 분석부터 설계, 구현, 테스트, 운영 대응까지 전체 흐름을 맡았습니다.  
중요하게 본 것은 기능 추가 자체보다, 이후 변경이 들어왔을 때 수정 지점이 분리되어 있고 장애가 났을 때 흐름을 따라가며 원인을 좁힐 수 있는 형태로 만드는 일이었습니다.

상태 전이 구조, 재시도/타임아웃 처리 방식, 로그 설계, 모듈 구성과 같은 기술 상세는 [Architecture](./Architecture.md) 문서에 정리했습니다.
